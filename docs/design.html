Nanotrace Iceberg Lakehouse Design

Apache Iceberg is the durable event lakehouse. ClickHouse is the serving plane: fast, rebuildable, freshness-checked, and fed from committed lakehouse snapshots.

Principles Architecture Ingest Loader Storage Materialization Queries Security Deployment

1. Design Principles

Lakehouse first Every loaded batch is committed to an Apache Iceberg V2 table before it is considered durable for serving.
Rebuildable serving ClickHouse tables are optimized serving indexes. They can be refilled from Iceberg commit metadata and Parquet data files.
Freshness checked Reads against raw and promoted serving tables are guarded by lakehouse serving watermarks unless the caller explicitly accepts stale data.

The deployed system writes accepted event batches to Iceberg, records the committed snapshot in ClickHouse metadata, then advances ClickHouse serving rows and watermarks. The canonical event shape stays raw JSON plus a small set of materialized routing columns; repeated analytical paths are promoted into narrow serving tables. This avoids the endless-widening-table problem while keeping all historical data replayable from Iceberg.

3. Ingest Path

Accepted events are staged locally, handed to the loader through S3/SQS, committed to Iceberg, then exposed through ClickHouse serving tables.
  1. Accept. The server receives batches through /v1/events, authenticates the API key/session, validates event structure, and materializes tenant identity.
  2. Stage. Events are written to local NDJSON part files through concurrent writer lanes. The HTTP path is decoupled from ClickHouse availability.
  3. Upload. Closed parts move through .ready, .uploading, and .done. Upload-stage processors may transform the part before it is PUT to S3.
  4. Notify. S3 object-created notifications feed SQS. Loader instances share that queue.
  5. Commit. The loader acquires the deterministic ingest ledger row, writes canonical Parquet data files, commits an Iceberg snapshot, and stores the full data-file list in lakehouse metadata.
  6. Serve. The loader posts FORMAT JSONEachRow batches to ClickHouse and advances observatory.serving_watermarks.

Durability model

Ingest transport is object-store centered, but analytical durability is lakehouse centered. The local writer protects the request path from immediate process failures; S3 stores the immutable accepted NDJSON part and drives SQS delivery; Iceberg stores committed canonical Parquet snapshots. Delivery is at-least-once, so the Postgres ingest ledger, Iceberg source-batch id, and ClickHouse insert tokens make redelivery idempotent.

4. Lakehouse Loader and Materializer

Production uses the lakehouse path by default. The loader commits each batch to Iceberg first, then writes raw ClickHouse serving rows and metadata. Promoted tables are kept current by the separate materializer service, which reads observatory.lakehouse_commits and only processes snapshots ahead of each target table watermark.

5. Iceberg and ClickHouse Storage Model

Current schema source: deploy/clickhouse/schema.sql. Applying it creates the serving schema plus lakehouse commit/watermark metadata. Iceberg table creation is handled by the Rust lakehouse writer.

Lakehouse table

The Iceberg table is nanotrace.events. It stores canonical columns such as event id, tenant id, timestamps, routing fields, source byte pointers, and the original payload as a JSON string column. Files are Parquet with ZSTD compression. Snapshot properties include record count, content hash, full data-file list, and deterministic nanotrace.source-batch-id for replay/dedupe.

Iceberg table: events

Canonical fact table

observatory.events is a rebuildable ClickHouse serving table partitioned by day and ordered by (tenant_id, timestamp, event_type, event_id). It stores event identity, timestamps, S3 source byte pointers, and a typed ClickHouse JSON payload. A small set of hot routing columns is materialized from JSON because they participate in sorting, filtering, or bloom indexes. 

CREATE TABLE observatory.events (
  event_id String,
  timestamp DateTime64(3, 'UTC'),
  source_file String,
  source_offset UInt64,
  source_length UInt32,
  data JSON(...),
  tenant_id LowCardinality(String) MATERIALIZED ifNull(data.tenant_id, ''),
  event_type String MATERIALIZED ifNull(data.event_type, ''),
  trace_id String MATERIALIZED ifNull(data.trace_id, ''),
  span_id String MATERIALIZED ifNull(data.span_id, ''),
  signal LowCardinality(String) MATERIALIZED multiIf(...),
  INDEX idx_event_id event_id TYPE bloom_filter(0.01) GRANULARITY 4,
  INDEX idx_trace_id trace_id TYPE bloom_filter(0.01) GRANULARITY 4,
  INDEX idx_span_id span_id TYPE bloom_filter(0.01) GRANULARITY 4
) ENGINE = MergeTree
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (tenant_id, timestamp, event_type, event_id);

Why JSON-first

  • The event payload remains open-ended enough for product analytics, observability data, and tenant-specific attributes.
  • Hot paths in the JSON type declaration get columnar subcolumns without forcing every path into a top-level table column.
  • New user attributes can be queried from raw data immediately, then promoted later through explicit SDK-managed or user/admin definitions.
  • Default write fanout is bounded to the core analytics pack; additional fanout is controlled by definitions instead of by arbitrary event shape.
  • Flattening every JSON path into a column would make the schema and write path widen forever; the final design keeps flexible raw payloads and promotes only paths that earn a serving model.

6. Materialization Control Plane

A full analytics system needs optimized serving tables. Current production has a continuous materializer for definition-backed field_index, event_measures, metric rollups, entity_state_updates, summary, trace summary, and retention report_results, sequence_report_results, and cohort_memberships. It reads shared lakehouse commit metadata from ClickHouse, scans only Iceberg snapshots ahead of the target watermarks, and advances only the serving tables it caught up. Full published-version report jobs still use the control-plane schema described here.

Lifecycle

  1. Define. SDK-managed definitions and user/admin definitions explicitly declare promoted fields, measures, reports, cohorts, funnels, and alert rollups.
  2. Plan. The current materializer plans from serving_watermarks. Future published-report executors create materialization_jobs and bounded materialization_chunks for historical or incremental windows.
  3. Build. Workers claim chunks with leases, write target serving tables, and update row/byte/error progress.
  4. Publish. materialization_versions marks completed target versions. Readers can select completed versions instead of partial writes.
  5. Continue. materialization_watermarks tracks high-water marks for incremental refresh lag.

Idempotency model

Iceberg commits remain append-only. Derived outputs are keyed by tenant, source snapshot/sequence, target id, target version, bucket/window, and deterministic dimension or entity identity. Retried jobs use deterministic ClickHouse insert tokens and rewrite the same logical target version. Serving queries prefer completed versions or caught-up watermarks and avoid reading partially built versions.

Implemented now: lakehouse commit replay, raw rebuild, incremental field/measure/metric/state materialization, summary report materialization, trace summary report materialization, retention report materialization, sequence report materialization, cohort membership materialization, continuous materializer service, and serving freshness guards. Still future work: full published-version report jobs and a full Iceberg REST/object-store production catalog.

7. Query and UI Behavior

Default logs view

With no grouping and no restrictive filter, the UI uses event_density_1s for the histogram and events for the event table. The flamegraph area is still present, but without a selected group it shows histogram-only density.

Group-by and filters

Core group lists read field_topk_1m, grouped histograms read field_density_1s, and core flamegraphs read flamegraph_rollups_1m. Exact id lookups can use field_values or the native event indexes. Non-core fields still require promotion/materialization before they are treated as guaranteed-fast group-bys.

Schema and report promotion

Schema/report definitions are the mechanism for making a repeated query shape fast. A definition identifies a tenant, a path or measure, extraction mode, and versioned config. Historical work should be represented as materialization jobs and chunks; completed outputs should be published as materialization versions. The lakehouse event loader stays focused on durable commits and baseline serving rows; promotion work belongs to materializers.

Serving freshness

Query APIs inspect table sources and compare them to observatory.serving_watermarks. Reads from events, field_index, event_measures, and entity_state_updates are rejected if the serving table is behind the latest lakehouse commit, unless the request sets allow_stale_serving=true.

8. Auth and Security

Authentication

  • Browser login uses one-time magic links. Tokens are random, hashed before storage, TTL-bound, and deleted on use.
  • Sessions are server-side hashed and stored in Postgres. Cookies are HttpOnly, SameSite=Lax, and secure when served over HTTPS.
  • API keys use an ntak_ prefix. Only hashes and short prefixes are stored; the full key is shown only at creation.
  • Roles and permissions separate ingest, query, schema/report writes, API key management, and admin actions.

Tenant isolation

Tenant identity is not accepted from arbitrary query input. The read service rewrites allowed table references into tenant-filtered subqueries using {__nanotrace_tenant_id:String}. The canonical table exposes materialized tenant_id, event_type, trace_id, span_id, and signal for efficient tenant-scoped reads.

Query sandbox

  • Only read-only SQL is accepted; multi-statement DDL/DML paths are rejected.
  • Tables are allowlisted to the current analytics schema: events, lakehouse_commits, serving_watermarks, field_index, field_values, field_density_1s, field_topk_1m, flamegraph_rollups_1m, event_density_1s, definitions, event_measures, measure_rollups, entity_state_updates, report_results, sequence_report_results, cohort_memberships, definition_stats, query_usage, materialization_jobs, materialization_chunks, materialization_versions, materialization_watermarks, and pipeline_metrics.
  • User values are passed as ClickHouse parameters; identifier-like names are validated.
  • ClickHouse requests are sent with readonly and bounded execution/result settings.
