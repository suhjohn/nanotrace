Nanotrace — Engineering Architecture

Tenant-keyed observability ingest pipeline. One /events API, object storage, a message queue, a Rust loader, and ClickHouse, deployed on AWS via Pulumi.

Overview Storage model Auth & security Deployment

1. System overview

Nanotrace ingests JSON events through an HTTP API, durably stages them across multiple concurrent writer lanes into local NDJSON part files, uploads closed parts to object storage, then loads each part into ClickHouse via storage queue notifications. The data model is tenant-keyed end-to-end: every write materializes a tenant_id and every read injects it into the query plan. Reads go through a stateless query service backed by ClickHouse for aggregates and object storage for raw event byte-range lookup.

Solid lines show the primary flow; dashed line is the on-demand byte-range event fetch.

Write path

Write path: validate → durable NDJSON → object storage → queue → loader → ClickHouse. File states shown below.
  1. Accept. Server validates each event against the schema-shaped JSON contract; rejects malformed payloads with 400.
  2. Stage. Accepted events are round-robin dispatched to one of NANOTRACE_WRITER_LANES (default 4) concurrent writer lanes, each backed by its own NDJSON part file on an XFS-formatted EBS volume. The writer flushes (fsyncs) at NANOTRACE_WRITER_FLUSH_BYTES (default 1 MiB) or NANOTRACE_WRITER_FLUSH_INTERVAL_MS (default 10 ms), whichever comes first. The HTTP 200 is returned after the event is committed to the lane queue. Parts rotate to .ready when they reach NANOTRACE_PART_MAX_BYTES (default 64 MiB) or NANOTRACE_PART_MAX_AGE_SECS (default 1 second).
  3. Upload. The background uploader picks up .ready files, renames them to .uploading, optionally runs upload-stage processors, and PUTs them to object storage. The S3 key layout is Hive-partitioned: <prefix>/dt=YYYY-MM-DD/hour=HH/host=<hostname>/lane=<id>/part-<timestamp_ms>.ndjson. On success the local file transitions to .done; on failure it becomes .failed.
  4. Notify. A bucket notification fans object creation events into the loader message queue.
  5. Load. The loader fetches each object, decodes lines, derives facet rows, and inserts via the ClickHouse HTTP JSONEachRow endpoint into observatory.events, observatory.field_counts_5m, and observatory.field_index.

Read path

  • POST /query on nanotrace-query compiles a constrained query against observatory.events with tenant scoping pinned to the caller's identity.
  • GET /events/{event_id} resolves the event's source_file, source_offset, and source_length from ClickHouse, then issues a ranged object storage GET to return the original line.

Sidecar / SDK

The UDP sidecar binds 127.0.0.1:4319, batches events, and POSTs to /events so the application thread never blocks on the ingest path. SDKs (packages/sdk-js, packages/sdk-py) call the same HTTP API with an API key.

2. Storage model

All semantic fields live inside a typed JSON column on observatory.events. Since ClickHouse 25.3, typed JSON paths are stored as physical columnar subcolumns. A field is only promoted to a top-level MATERIALIZED column when it must participate in ORDER BY, carry a skip index, or use a non-default codec.

Tables

TableEnginePurpose
observatory.events MergeTree Canonical event store. ORDER BY (tenant_id, event_type, timestamp, trace_id, span_id); partitioned by day. Bloom-filter indexes on trace_id, span_id, event_id.
observatory.field_counts_5m SummingMergeTree Pre-aggregated counts per (tenant_id, key, value, bucket_time) for fast facet listings.
observatory.field_index MergeTree Facet → event lookup index. Lets the UI jump from a value chip to matching events without scanning events.
observatory.definitions ReplacingMergeTree Per-tenant configuration of which JSON paths are tracked for facets. Refreshed by the loader.

Why JSON-first

  • Open schema for OTEL semantic conventions, product analytics, and exotic per-tenant fields without DDL changes.
  • Hot paths get type hints (e.g. http.method LowCardinality(Nullable(String))) so they compress and scan as columnar data.
  • Sort-key columns are MATERIALIZED ifNull(data.x, '') — columns in ORDER BY cannot be nullable, but the JSON path stays nullable so absent vs empty stays distinguishable.
  • signal is a derived multiIf classifier so trace / metric / log / analytics rows can be filtered without inspecting event_type string-by-string.

3. Auth & security

Authentication

  • Browser login uses one-time magic links sent via AWS SES. Tokens are 256-bit random, SHA-256 hashed before storage in Postgres, and expire after NANOTRACE_MAGIC_LINK_TTL_SECS (default 10 min). Deleted on use to prevent replay.
  • Sessions are SHA-256 hashed server-side. Cookies are HttpOnly, SameSite=Lax, and Secure when the public base URL is HTTPS. Default TTL is 7 days.
  • API keys use ntak_… prefix with 32 bytes of random entropy. Only the SHA-256 hash and a short prefix are stored; the full key is revealed once at creation. Keys carry a role (Viewer, Service, Admin) and scoped permissions (ingest:write, query:read, facets:write, etc.), with optional expiration and revocation.
  • Allowed emails are configured via NANOTRACE_ALLOWED_EMAILS / NANOTRACE_ADMIN_EMAILS (literal addresses, *@domain globs, or /regex/).

Tenant isolation

The data model is tenant-keyed end-to-end. tenant_id is the first column in every ClickHouse ORDER BY key. At query time, the auth layer injects a WHERE tenant_id = {__nanotrace_tenant_id:String} clause into every SQL plan — the parameter value comes from the caller's authenticated identity, never from user input. Every ingested event row is materialized with the same tenant key. A single deployment can carry several logical tenants behind one auth domain.

Query sandbox

  • Only SELECT / WITH statements allowed; DDL, DML, and multi-statement queries are rejected.
  • Table references are validated against a hard-coded whitelist (events, field_counts_5m, field_index, definitions). Access to system.* or other tables is blocked.
  • User values are passed as ClickHouse query parameters, not interpolated into SQL. Parameter names are validated as identifiers; only scalar types are accepted.
  • ClickHouse queries execute with readonly=1, max_execution_time=30s, max_result_rows=100000, and max_bytes_to_read=1GB.
  • SQL comments are stripped before analysis to prevent obfuscation.

Encryption

  • At rest. S3 objects use SSE with a configurable KMS key or AES-256 default. EBS volumes and RDS storage are encrypted. ClickHouse Cloud manages its own encryption.
  • In transit. TLS terminates at the ALB (ACM cert) or at the edge (Cloudflare). Internal traffic to S3, SQS, and SES uses the AWS SDK over TLS. RDS connections are within private subnets.

Network & infrastructure

  • VPC isolation. Per-stack VPC (10.42.0.0/16). ALB SG accepts 80/443 from the Internet; instance SG accepts the app port only from the ALB SG; Postgres SG accepts 5432 only from the instance SG.
  • IMDSv2 enforced. httpTokens=required on all EC2 instances to prevent SSRF-based credential theft.
  • IAM least-privilege. Instance role is scoped to specific S3 prefixes, the loader SQS queue, the ECR repository, SES sending, and the data KMS key. No wildcard resources.

4. Deployment

One Pulumi program (deploy/pulumi/nanotrace) provisions a self-contained stack per deployment: VPC, ALB, two EC2 Auto Scaling Groups, S3, SQS, RDS Postgres, ECR, CloudFront, SES, and IAM. External dependencies are AWS credentials, ClickHouse connection settings, and DNS ownership.
