Nanotrace Current System Design

Raw-first, tenant-scoped event analytics on ClickHouse. The ingest hot path writes one canonical event fact table by default; every additional analytical read model is explicit, schema-defined, and backfillable.

Principles Architecture Ingest Loader ClickHouse Materialization Queries Security Deployment

1. Design Principles

Raw first observatory.events is the canonical fact table. It stores accepted events exactly once in the default loader mode.
Explicit acceleration Core group-by rollups are created by default; additional fields, measures, entity states, cohorts, and reports become derived tables after promotion.
Tenant scoped Every table is keyed by tenant_id. The read layer injects tenant filters from authenticated identity.

The current system writes one canonical raw event table and a bounded default analytics pack for core group-by, histogram, flamegraph, and exact-id lookup paths. Arbitrary JSON fields are still raw-first until promotion/materialization. The schema also includes the control-plane tables needed for a full analytics system: query usage, recommendations, materialization jobs, chunk status, completed versions, and watermarks. The actual user-driven promotion executor remains separate from the raw loader hot path.

3. Ingest Path

Accepted events are durably staged locally before S3 upload. S3 is the handoff boundary to the loader.
  1. Accept. The server receives batches through /events, authenticates the API key/session, validates event structure, and materializes tenant identity.
  2. Stage. Events are written to local NDJSON part files through concurrent writer lanes. The HTTP path is decoupled from ClickHouse availability.
  3. Upload. Closed parts move through .ready, .uploading, and .done. Upload-stage processors may transform the part before it is PUT to S3.
  4. Notify. S3 object-created notifications feed SQS. Loader instances share that queue.
  5. Load. The loader fetches objects and posts FORMAT JSONEachRow batches to ClickHouse using async insert.

Durability model

Ingest durability is object-store centered. The local writer protects the request path from immediate process failures; S3 stores the immutable accepted NDJSON part; SQS drives eventual loading. Delivery is at-least-once, so downstream tables and insert tokens must tolerate retry. The system optimizes for not dropping events over exactly-once visibility.

4. Loader Modes

The loader has two intentional derivation modes today. The production default is raw; this inserts only observatory.events. The optional promoted mode inserts raw events plus definition-backed rows into secondary tables, but the final analytics architecture keeps that work outside the raw loader path. Promotion, report refresh, cohort construction, and sequence materialization should be performed by separate workers using the job tables described below.

ModeWritesUse case
raw events Default high-throughput ingest. No per-event field fanout except ClickHouse materialized columns and materialized views already attached to events.
promoted events, field_index, event_measures, entity_state_updates Acceleration for explicitly promoted business analytics fields, numeric measures, and entity state changes.

Definition refreshes are cached for one minute by default. Raw mode does not need definitions and skips the definition query entirely. Promoted mode reads enabled definitions from ClickHouse and derives only the configured paths.

5. ClickHouse Storage Model

Current schema source: deploy/clickhouse/schema.sql. Applying it creates the current raw-first schema and drops obsolete write-time materialized views and tables from previous designs.

Canonical fact table

observatory.events is a MergeTree table partitioned by day and ordered by (tenant_id, timestamp, event_type, event_id). It stores event identity, timestamps, S3 source byte pointers, and a typed ClickHouse JSON payload. A small set of hot routing columns is materialized from JSON because they participate in sorting, filtering, or bloom indexes. 

CREATE TABLE observatory.events (
  event_id String,
  timestamp DateTime64(3, 'UTC'),
  source_file String,
  source_offset UInt64,
  source_length UInt32,
  data JSON(...),
  tenant_id LowCardinality(String) MATERIALIZED ifNull(data.tenant_id, ''),
  event_type String MATERIALIZED ifNull(data.event_type, ''),
  trace_id String MATERIALIZED ifNull(data.trace_id, ''),
  span_id String MATERIALIZED ifNull(data.span_id, ''),
  signal LowCardinality(String) MATERIALIZED multiIf(...),
  INDEX idx_event_id event_id TYPE bloom_filter(0.01) GRANULARITY 4,
  INDEX idx_trace_id trace_id TYPE bloom_filter(0.01) GRANULARITY 4,
  INDEX idx_span_id span_id TYPE bloom_filter(0.01) GRANULARITY 4
) ENGINE = MergeTree
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (tenant_id, timestamp, event_type, event_id);

Why JSON-first

  • The event payload remains open-ended enough for product analytics, observability data, and tenant-specific attributes.
  • Hot paths in the JSON type declaration get columnar subcolumns without forcing every path into a top-level table column.
  • New user attributes can be queried from raw data immediately, then promoted later if repeated query cost justifies a read model.
  • Default write fanout is bounded to the core analytics pack; additional fanout is controlled by definitions instead of by arbitrary event shape.

6. Materialization Control Plane

A full analytics system needs optimized serving tables. Core group-by, histogram, flamegraph, and exact-id lookup tables are populated by ClickHouse materialized views from raw events. User-promoted fields and reports should be populated by controlled jobs rather than by arbitrary loader fanout. The schema has the records needed to plan, track, retry, and serve those additional materializations without implementing the promotion executor yet.

Lifecycle

  1. Observe. query_usage records query fingerprints, JSON paths, raw fallback status, latency, and scan cost.
  2. Recommend. optimization_recommendations stores proposed fields, measures, reports, cohorts, funnels, and alert rollups.
  3. Plan. A future executor creates materialization_jobs and bounded materialization_chunks for historical or incremental windows.
  4. Build. Workers claim chunks with leases, write target serving tables, and update row/byte/error progress.
  5. Publish. materialization_versions marks completed target versions. Readers can select completed versions instead of partial writes.
  6. Continue. materialization_watermarks tracks high-water marks for incremental refresh lag.

Idempotency model

Raw events remain append-only. Derived outputs are keyed by tenant, target id, target version, bucket/window, and deterministic dimension or entity identity. Retried jobs should rewrite the same logical target version. Serving queries should prefer completed versions from materialization_versions and avoid reading partially built versions.

This pass creates the schema and documentation for the materialization control plane. It does not implement automatic promotion, recommendation generation, job claiming, report execution, cohort execution, or sequence execution.

7. Query and UI Behavior

Default logs view

With no grouping and no restrictive filter, the UI uses event_density_1s for the histogram and events for the event table. The flamegraph area is still present, but without a selected group it shows histogram-only density.

Group-by and filters

Core group lists read field_topk_1m, grouped histograms read field_density_1s, and core flamegraphs read flamegraph_rollups_1m. Exact id lookups can use field_values or the native event indexes. Non-core fields still require promotion/materialization before they are treated as guaranteed-fast group-bys.

Schema and report promotion

Schema/report definitions are the mechanism for making a repeated query shape fast. A definition identifies a tenant, a path or measure, extraction mode, and versioned config. Historical work should be represented as materialization jobs and chunks; completed outputs should be published as materialization versions. The raw loader should remain independent from this lifecycle.

8. Auth and Security

Authentication

  • Browser login uses one-time magic links. Tokens are random, hashed before storage, TTL-bound, and deleted on use.
  • Sessions are server-side hashed and stored in Postgres. Cookies are HttpOnly, SameSite=Lax, and secure when served over HTTPS.
  • API keys use an ntak_ prefix. Only hashes and short prefixes are stored; the full key is shown only at creation.
  • Roles and permissions separate ingest, query, schema/report writes, API key management, and admin actions.

Tenant isolation

Tenant identity is not accepted from arbitrary query input. The read service rewrites allowed table references into tenant-filtered subqueries using {__nanotrace_tenant_id:String}. The canonical table exposes materialized tenant_id, event_type, trace_id, span_id, and signal for efficient tenant-scoped reads.

Query sandbox

  • Only read-only SQL is accepted; multi-statement DDL/DML paths are rejected.
  • Tables are allowlisted to the current analytics schema: events, field_index, field_values, field_density_1s, field_topk_1m, flamegraph_rollups_1m, event_density_1s, definitions, event_measures, measure_rollups, entity_state_updates, report_results, sequence_report_results, cohort_memberships, definition_stats, query_usage, optimization_recommendations, materialization_jobs, materialization_chunks, materialization_versions, materialization_watermarks, and pipeline_metrics.
  • User values are passed as ClickHouse parameters; identifier-like names are validated.
  • ClickHouse requests are sent with readonly and bounded execution/result settings.
