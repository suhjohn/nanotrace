Nanotrace System Design

Concise current-state spec for ingest, storage, query planning, arbitrary KV filtering, materialization, replay, and validation. The implementation is raw-first: preserve arbitrary event JSON, index bounded scalar KVs for immediate filtering, and materialize only SDK-typed or user-defined semantics.

TruthIceberg stores accepted normalized events and is the replay/rebuild base.
ServingClickHouse stores raw rows, generic indexes, and derived read models.
IngestHTTP accepts to Kafka; the normalizer owns validation, Iceberg commits, invalid rows, and offset progress.
SemanticsOnly SDK types or explicit definitions create rollups, funnels, cohorts, reports, or promoted facets.
Core rule: all bounded scalar KVs are filterable by default through event_kv_index. They do not become grouping dimensions, measures, cohorts, funnels, reports, or dashboards unless the SDK or a user/admin definition says so.

ConcernCurrent contract
Accepted writePOST /v1/events returns 202 after Kafka accepts the raw request body. It is not a visibility guarantee.
VisibilityRows become queryable after the normalizer commits valid events to Iceberg and the materializer catches up ClickHouse events, event_text_index, event_search_terms, and event_kv_index.
ReplayKafka processing is at-least-once; deterministic source batch ids and materializer insert tokens keep replay idempotent for tested paths.
Tenant isolationThe authenticated API key determines tenant identity; client-supplied tenant_id is overwritten. Structured product queries are tenant-scoped; cross-tenant analysis belongs in operator-controlled lakehouse query/export tooling, not a server API route.
High cardinalityHigh-cardinality strings are filterable, not automatically promoted. Grouping/rollup requires an explicit definition and guardrails.

Event Contract

Clients send one event object or an array of event objects. Paths used by filters and definitions are paths inside data.

{
  "event_id": "evt_123",
  "timestamp": "2026-06-04T23:00:00.000Z",
  "data": {
    "event_type": "track",
    "name": "Checkout Completed",
    "plan": "pro",
    "route": "/checkout",
    "latency_ms": 175,
    "request_id": "req_123",
    "entities": {
      "user_id": "usr_1",
      "account_id": "acct_1",
      "session_id": "sess_1"
    },
    "llm": { "model": "gpt-4.1" }
  }
}
KindTypical fieldsSemantic effect
Track/productevent_type=track, name, entity ids, arbitrary KVsStored raw and indexed for filters. No automatic rollup.
Trace/spanspan_start, span_end, trace_id, span_id, duration_msSignal becomes trace. Span fields are not auto-promoted as measures.
Metricevent_type=metric, metric_name, metric_type, metric_valueSDK-managed metric definition can materialize metric rollups.
Logevent_type=log, severity, message, serviceStored raw; exact bounded KVs are indexed and ranked interactive search uses event_search_terms.
Nested KVs: objects flatten to dot paths such as llm.model. Scalar arrays emit repeated rows at the same path. Arrays of objects use paths such as items[].sku and carry scope_path/scope_index so filters can require values from the same array element.

Queries

Question shapePreferred substrateNotes
Latest events, event detail, bounded trace/span lookupevents, field_valuesTenant and time predicates are mandatory for broad reads.
Bounded event text searchevent_text_indexEvent-level text filters use a capped search document, not arbitrary full-text search.
Ranked interactive searchevent_search_terms/v1/query with type=search ranks events by matched weighted exact, prefix, or bounded fuzzy terms and joins back to events.
Arbitrary scalar equality, IN, exists, numeric rangesevent_kv_indexCorrect default for unpromoted paths such as request_id, plan, llm.model.
Repeated grouping/facetingfield_rollups for core fields; field_index for explicit definitionsHigh-cardinality group-by needs fanout checks.
Percentiles/sums/averagesmeasure_rollups or event_measuresRequires explicit measure or SDK metric semantics.
Latest/as-of entity stateentity_state_current, entity_state_updatesLatest reads use the compact snapshot; as-of reads use update history.
Funnel conversionsequence_report_resultsRequires a sequence definition with entity id and steps.
Reusable populationcohort_membershipsUse cohorts for retention denominators, experiments, eligibility, and repeated drilldowns.
Large text search, regex over arbitrary JSONNot event_kv_indexUse event_search_terms for ranked exact, prefix, or bounded fuzzy term search and NANOTRACE_LAKEHOUSE_QUERY_REGEX for offline regex scans; advanced analyzers and semantic search remain separate.

POST /v1/query is the structured product query API. The request body carries a type such as events, search, measure, funnel, cohort, report, state, or alerts. Event queries plan arbitrary scalar filters through event_kv_index and use promoted indexes only with freshness proof.

There is no public server SQL route. POST /v1/query is the user-facing structured query API and remains tenant-scoped. SQL-shaped historical analysis should use operator-controlled lakehouse export/query tooling.

Production deploys require Kafka plus a shared Iceberg REST catalog and object-store warehouse. Local filesystem lakehouse storage is for local or single-node operation only.

Boundaries And Open Work

StatusItem
CurrentDefault ingest is HTTP -> Kafka -> normalizer -> Iceberg -> materializer -> ClickHouse.
CurrentArbitrary bounded scalar KVs are filterable through event_kv_index.
CurrentQuery responses carry planner/source metadata and per-filter event routes; the event explorer shows these in the event-table plan badge.
CurrentLakehouse rebuild/materializer supports fields, measures, metric rollups, states, reports, sequences, and cohorts.
Current splitThe definitions API handles field, measure, rollup, and state backfills. Reports, sequences, and cohorts are produced by the lakehouse materializer, not by synchronous API backfill.
CurrentThe materializer publishes completed materialization_jobs, materialization_chunks, materialization_versions, and materialization_watermarks rows for produced report, sequence, and cohort outputs.
Current/v1/definitions/{definition_id}/backfills creates tenant-scoped report, sequence, and cohort jobs/chunks; /v1/backfills lists them, and the queued materialization executor claims pending chunks for explicit historical backfills.
CurrentThe schema UI can create reviewed field, measure, state, saved search, custom report, sequence, cohort, and alert definitions from JSON templates, then backfill field/measure/state definitions, queue report/sequence/cohort backfill jobs for the selected source window, or hand hot event-match alerts to nanotrace-alerts.
Currentnanotrace-alerts supports hot event-match alerts plus webhook outbox delivery through alert_events and alert_notifications.
Not currentNo processor-management API/runtime is part of the default architecture.
CurrentThe event explorer exposes ranked exact-token, prefix, bounded fuzzy, and phrase search controls; exact/prefix/fuzzy search is served by event_search_terms, while exact KV filtering remains separate from search ranking.
Not currentDashboard-specific APIs are removed for now. Reusable views should use structured /v1/query helpers for events, measure, state, report, funnel, cohort, and alert reads.
CurrentThe event explorer can approve planner recommendations into field, saved search, numeric measure, and basic summary report definitions; grouped report approval also queues the backfill job for the viewed source window and shows matching job status.
CurrentGET /v1/query/recommendations exposes recent query-usage recommendation records, and the schema UI can prefill field, measure, and grouped report drafts from that feed while showing live approval, backfill, and report-job status.
Not currentAdvanced analyzers, semantic search, and regex indexing are not exact KV-index features; regex exists only as an offline lakehouse query/export filter.
