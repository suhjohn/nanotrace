Facet vs Correlation Index Scaling

The hard part is not storing arbitrary fields. The hard part is choosing which fields deserve fast browseable group-by behavior, which fields only need exact lookup, and which fields should stay in the raw event body.

One Sentence Model

Facets answer discovery questions. "What values exist, how many events do they have, and which groups are hottest?" Correlation keys answer lookup questions. "I already have this ID. Show me the related events."

Facet Table

A facet table is optimized for browsing, grouping, and drilling into a selected dimension. It usually has two pieces: 

event_facets
  tenant_id, key, value, bucket_time, count, error_count

event_facet_index
  tenant_id, key, value, timestamp, event_id, metadata...

The summary table makes top groups cheap. The membership table makes "open this group" cheap.

Correlation Table

A correlation table is optimized for exact lookup by known IDs. It is not meant to power the whole group sidebar. 

event_correlation_index
  tenant_id, key, value, timestamp, event_id, metadata...

It is best for keys like trace_id, request_id, span_id, and event_id.

The Shape Looks Similar, But The Workload Is Different

Question Example Best Data Structure Why

Cardinality Scenarios

Low 10s of values. Example: service, env, status_code
Medium 100s to 100k values. Example: http.route, model, workspace_id
High Millions of values. Example: user_id, session_id, account_id
Key Type Examples Facet Behavior Correlation Behavior Recommendation
Low cardinality service, environment, signal, http.status_code Excellent. Summary rows are compact, group-by is cheap, and drilldown is useful. Works for exact lookup, but misses the main value: top lists and aggregate navigation. Make these default facets.
Medium cardinality http.route, name, model, provider Usually good if the field is used often. Write cost is meaningful but controlled. Good only if users paste a specific value. Weak for "show me top routes/models". Make common ones default. Let users add the rest as hot dimensions.
High cardinality user_id, account_id, session_id Expensive but sometimes worth it. Product and support workflows often need "top users by errors" and "open user X". Excellent for "show user X" if the value is known. Does not cheaply answer "top users". Treat as opt-in or carefully chosen default facets. Keep summaries bucketed and limit top-N reads.
Per event identity event_id, span_id, often request_id Bad default. Nearly every value is unique, so group lists become huge and not analytically useful. Excellent. Exact lookup is the actual workload. Keep out of default facets. Put in correlation index or use dedicated columns/indexes.
Trace identity trace_id, parent_span_id Usually poor as a browseable facet. You do not want a sidebar of millions of trace IDs. Strong. Trace view starts from a known trace ID or a clicked event and fetches related events. Use correlation lookup or promoted columns on events.

Write-Time Cost

Write cost is roughly proportional to how many derived rows the loader emits per event. If one event has values for 12 active facets, the generic facet membership index can write up to 12 extra rows.

Raw Event

Always one row in events. This is the source of truth.

->

Facet Writes

One membership row per active browseable key=value, plus summary rows.

->

Correlation Writes

Only a small fixed set of exact lookup IDs, usually 1 to 3 rows per event.

Read-Time Cost

Read performance depends on whether the physical table order matches the question. Exact lookup and top-N group-by want different physical layouts.

Exact Lookup Wants Value First

ORDER BY (tenant_id, key, value, timestamp)

This makes request_id=req_123 or trace_id=abc fast because ClickHouse can jump to a narrow range.

Group-By Wants Time Buckets Or Time First

event_facets:
ORDER BY (tenant_id, key, bucket_time, value)

This makes "top values in this time range" cheap because the query scans aggregated buckets instead of event membership rows.

Why User ID Is Different From Request ID

user_id

High cardinality, but product-support workflows justify it: top users by errors, open a user's activity, compare affected users, build cohorts.

This can be worth the facet cost if we intentionally support it and keep the query surface bounded.

request_id

Usually one request or a small cluster of events. Users normally paste it, click it, or arrive from an event detail.

It should be searchable, but a "top request IDs" sidebar is usually noise.

Recommended Nanotrace Policy

Class Keys Storage UI Behavior
Default facets service, environment, event_type, signal, name, http.route, http.method, http.status_code, severity_text, metric_name event_facets + event_facet_index Appear in group menu and sidebar by default.
Intentional high-cardinality facets user_id, account_id, session_id, selected customer fields event_facets + event_facet_index, only when active Can be enabled as hot dimensions. Backfill old data when enabled.
Correlation IDs trace_id, span_id, parent_span_id, request_id, event_id event_correlation_index or promoted columns on events Search box, clickable links, trace/request views. Not normal group sidebar values.
Cold arbitrary fields Everything else in data Raw JSON in events Ad hoc query works, but large-scale group-by requires promoting to hot dimension.

The Engineering Rule

Do not decide based only on cardinality. Decide based on the product question. If users need to discover values and compare groups, it is a facet. If users already have the value and want related events, it is correlation. If users rarely query it, keep it raw until they ask for it. 

Facet:
  "What values exist, how many, and which should I click?"

Correlation:
  "I have this ID. Show me what happened."

Raw event field:
  "Store it now. Optimize it only after it becomes a real workflow."
