Nanotrace Query Use Cases

A practical taxonomy for what runs now on the Iceberg-backed ClickHouse serving plane, what becomes fast after schema/report materialization, and what needs a new subsystem when the target load is 1M requests/sec and 100M events/sec.

1M req/s Target request rate. Raw scans must be rare, bounded, or point-lookups.
100 events/req Business and observability flows can emit many correlated events per request.
Iceberg source Serving tables are rebuildable from committed lakehouse snapshots and guarded by watermarks.

Classification

At this scale, Iceberg is the durable source of truth. ClickHouse raw and promoted tables are serving indexes, not the only copy of analytical history.

The load-test profile that best exercises this design is codex: it emits correlated agent workflows rather than isolated rows, including span start/end, planning, retrieval, LLM calls, tool calls, evaluation, safety, runtime metrics, processor events, and logs.

Group 1: run now

Uses default serving tables, indexes, and freshness watermarks already maintained by the system.

Group 2: promote/materialize/report

Has a clear optimization path through schema definitions, materialization jobs, and published result versions.

Group 3: new subsystem

Cannot be made efficient by the current schema/report materialization model alone.

Optimization Lifecycle

Group 2 means the serving model is known and schema-backed. The current system already has Iceberg commit tracking, raw rebuild, serving watermarks, and a continuous materializer for promoted fields, measures, and entity states. Recommendation generation and report/cohort/sequence executors remain separate implementation work.

Observe

query_usage records raw fallback query shapes, JSON paths, scanned bytes, latency, and result size.

Recommend

optimization_recommendations stores proposed fields, measures, reports, cohorts, funnels, and alerts.

Build

The current promoted-table materializer reads lakehouse_commits and serving_watermarks; broader executors use materialization_jobs and materialization_chunks.

Serve

serving_watermarks, materialization_versions, and materialization_watermarks let readers select caught-up serving data and observe lag.

Product Rule

Group 1

Run immediately.

Group 2

Create or select a definition/report, enqueue materialization work, then serve only completed versions instead of silently running a large raw scan.

Group 3

Present as unsupported, offline, or new-subsystem work, not as an interactive query.
