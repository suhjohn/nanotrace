A question-led reference for engineering telemetry and business analytics. Each numbered item starts with the user question, then shows SDK/spec-shaped instrumentation pseudocode, definitions when needed, how querying works, and the answer the user should see.

Use Case Form

The product-facing API should be one JSON endpoint: POST /v1/query. The request body carries a type such as events, search, measure, funnel, cohort, report, state, or alerts. That keeps one query surface while still letting the server validate inputs, enforce tenant scope, check freshness, and return stable response shapes. SQL is not the product workflow; historical SQL-shaped analysis belongs in operator-controlled lakehouse query/export tooling.

  1. Write the heading as the literal user question.
  2. Show the event or SDK calls the customer should emit. Prefer SDK-shaped pseudocode over table names.
  3. State whether this works from raw events and event_kv_index, or whether it needs a field, metric, measure cube, sequence, cohort, state, report, or alert definition.
  4. Show queries as POST /v1/query JSON requests with a type field. Do not teach SQL as the product workflow.
  5. Say what the user sees, including freshness or materialization state when relevant.

Integration Test Coverage Map

The Kafka integration test and loadtest-backed materialization check are the concrete product backbone for this document. They exercise tenant isolation, Kafka ingest, materializer-loaded ClickHouse serving rows, lakehouse commits, serving watermarks, materialization versions and watermarks, KV filters, nested arrays, field backfill, SDK metrics, measure cubes, funnels, cohorts, cohort-scoped measures, and load-generated raw plus derived serving catch-up.
