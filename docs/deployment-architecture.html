Nanotrace Deployment Architecture

Concrete deployment nodes for the current AWS-oriented Nanotrace shape: one public nanotrace-server instance can serve both gateway and control-plane routes, while dedicated organization data planes run the same server binary in internal-only mode.

One Public Server Binary

The public EC2 task runs nanotrace-server. In public mode, that one process serves customer routes, auth, organizations, API keys, facets, processors, dashboards, and static UI.

Same Binary, Different Mode

The data-plane EC2 task also runs nanotrace-server, but with NANOTRACE_DATA_PLANE_ORGANIZATION_ID set. It only exposes internal ingest/read routes plus health and metrics.

Raw Log Is Source of Truth

The data-plane server writes local EBS NDJSON first. S3 stores immutable raw parts, while ClickHouse indexes rows and source offsets for fast queries and exact event retrieval.

Request Paths

Hot path avoids S3 and ClickHouse before acknowledging ingest.

Ingest

SDK → public ALB → public nanotrace-server /events → data ALB → data-plane nanotrace-server /internal/events → EBS fsync → receipt. Upload, SQS, loader, and ClickHouse insertion happen afterward.

Query

UI/API → public ALB → public nanotrace-server /query → data ALB → data-plane nanotrace-server /internal/query → ClickHouse. The query layer constrains SQL and injects tenant scoping.

Event Detail

GET /events/{id} looks up the event pointer in ClickHouse, fetches the exact S3 byte range, validates the stored event_id, and returns the raw JSON.
